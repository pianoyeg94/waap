# Тестовое задание

## Контекст

Наш домен — security / WAAP. Один из примитивов детекционного слоя — матчинг большого набора сигнатур против потока HTTP-запросов. Задача — упрощённая модель этой работы: чистый perf-engineering.

## Задача

На N-процессорной системе сматчить корпус HTTP-запросов против ~12 000 regex-паттернов из `nmap-service-probes`. Цель — максимальный throughput (запросов/сек). Для каждого запроса выдать список сматчившихся имён сервисов с учётом разделения `match` / `softmatch` и приоритета probe.

## Входные данные

**Паттерны:**
```
https://raw.githubusercontent.com/nmap/nmap/master/nmap-service-probes
```
~2.5 MB, PCRE-подобный синтаксис. Грамматика описана в шапке файла. Несовместимость части паттернов с выбранным regex-движком приелема — для решения-обсуждения.

**HTTP-корпус (JSON):**
```
https://downloads.openappsec.io/waf-comparison-project/legitimate.zip
https://downloads.openappsec.io/waf-comparison-project/malicious.zip
```
1 040 242 legit + 73 924 malicious request'а с метаданными.

Финальный тест на ~1M.

**Что матчить:** для каждого запроса — нормализованный плоский байтовый поток из `method + URI + headers + body`. Способ нормализации — на твоё усмотрение.

## Что сдать

1. Код (Go или Rust на выбор)
2. Скрипт запуска и воспроизводимая инструкция «как получить числа»
3. Замеры:
   - Wall-clock на полный прогон
   - Throughput (req/s)
   - Линейность по ядрам
4. Baseline для сравнения: наивный однопоточный проход (твоего выбора движка) хотя бы на 100k запросов — нужно как нижняя планка для оценки speedup.
5. README (1–2 страницы):
   - Какие варианты пробовал, что отбросил и почему
   - Где упёрся (CPU? memory bandwidth...)
   - Как поступил с паттернами, несовместимыми с выбранным движком
   - Что бы делал на 100M / 1B и почему текущее решение туда без изменений не масштабируется
